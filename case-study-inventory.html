Logistics & Inventory API

Engineering a mission-critical backend service capable of sustaining high-throughput logistics routing and warehouse tracking.

1. Project Overview

A multi-national 3PL (third-party logistics) provider required a complete rewrite of their aging order management backend. The previous monolith was suffering from severe race conditions during concurrent stock updates, leading to catastrophic warehouse desynchronization during peak holiday periods.

2. Technical Challenges

The API needed to solve foundational data integrity issues:

  • Concurrency & Race Conditions: Preventing overselling when hundreds of client applications attempt to reserve the same SKU simultaneously.
  • Query Performance: Resolving heavily nested relational queries (Orders -> Shipments -> Warehouses -> Pallets).
  • System Interoperability: Creating a unified access layer that cleanly integrates with dozens of legacy external vendor APIs.

3. Engineered Solution

I led the migration to a microservices-adjacent architecture powered by the NestJS framework. To solve the complex querying constraints, I implemented a GraphQL API using Apollo Server, allowing front-end teams to fetch precise nested trees of data without encountering over-fetching typical of REST.

To eliminate the crippling race conditions, I established strict ACID-compliant transaction boundaries within a deeply indexed PostgreSQL database. For highly contested, fast-moving SKUs, I introduced a distributed locking mechanism via Redis, reliably enforcing serial access to inventory decrements and completely halting accidental oversells.
